# GoRipper v0.0.3-pre — LEA Length Inference + Concatenated URL Splitting + First Test Suite

**Release date:** 2026-03-03

### Install

```bash
go install github.com/muxover/goripper/cmd/goripper@v0.0.3-pre
```

Or download a pre-built binary from [Releases](https://github.com/muxover/goripper/releases/tag/v0.0.3-pre).

---

## Summary

This release ships two complementary string-quality fixes: LEA-guided strings now attempt to
infer their exact length from nearby instructions, and a new post-classification pass splits
any remaining URL blobs that contain multiple embedded URLs (the CMOVNE compiler pattern). The
result: consecutive hardcoded URLs — such as four Discord webhook URLs packed back-to-back in
`.rodata` — now appear as four clean individual strings instead of one dirty 484-byte blob.

Also included: the first automated test suite (14 tests covering unit and integration paths)
and two small defensive fixes to type recovery and pclntab false-positive handling.

The LEA-guided extraction itself was shipped earlier in commit `633d054` and is part of this
release.

## What's new

### Fix 1 — LEA string length inference

**Before:** When `CrossReference()` found a LEA target not already in the header-pair scan,
it emitted a 512-byte printable run from that address. Adjacent string literals packed without
separators in `.rodata` were concatenated into one dirty blob (e.g. four Discord webhook URLs
appearing as a single 484-byte string).

**After:** The scanner now records the byte position of each LEA instruction. In the second
pass it decodes up to 15 instructions forward from that position looking for a `MOV reg, <imm>`
where the immediate falls in `[minStringLen, 4096]`. When found, that value is used as the
exact string length. Otherwise it falls back to the 512-byte run.

This eliminates dirty blobs for the common Go compiler pattern:

```asm
LEA RAX, [RIP+offset]   ; string pointer
MOV RCX, 42             ; string length  ← now detected
```

Strings using the CMOVNE conditional-select pattern (fasthttp, some stdlib code) still fall
back to the 512-byte run when the length cannot be inferred from the window. Those cases are
handled by Fix 1b below.

### Fix 1b — Concatenated URL splitting (CMOVNE residual)

**Problem:** When four Discord webhook URLs are stored consecutively in `.rodata` without
separators, and the Go compiler uses a CMOVNE conditional select, `findLengthNearby()` either
finds no MOV or finds the wrong one — the 512-byte fallback fires and emits a 484-byte blob
containing all four URLs.

**Fix:** `SplitConcatenatedURLs()` runs as a post-classification pass after `Classify()`. Any
URL-typed string that (a) starts with `https?://` and (b) contains at least one additional
`https?://` at a non-zero position is split at each URL boundary. The original blob is
replaced by its components; each component inherits the parent's `ReferencedBy` list and gets
an adjusted `Offset`.

**Result:**Four consecutive Discord webhook URLs now appear as four clean
individual entries, each attributed to the correct set of sending functions.

### Fix 1c — URL classifier overhaul

**Problem (old):** `classifyOne()` used `strings.Contains(s, "://")` as a fallback. Any
512-byte printable-run blob that happened to contain `://` somewhere in its 512 bytes was
classified as URL. A `--type url` query returned 31 entries of which 24 were
dirty concatenated error-message blobs, burying the real signals.

**Fix:**
1. Remove the `://` Contains fallback entirely. The URL type now only matches strings that
   **start** with a URI scheme (`reURL`).
2. Generalise `reURL` from the hardcoded list `https?|ftp|ws|wss` to the RFC 3986 pattern
   `[a-z][a-z0-9+\-.]*://` — any valid URI scheme is now recognised (mongodb, redis, git,
   ssh, ldap, …).

**Result:**`--type url` returns exactly 7 clean entries — 4 Discord webhook
endpoints and 3 Instagram API endpoints — with zero noise.

### Fix 2 — Commit 633d054: LEA-guided extraction (shipped earlier)

`CrossReference()` now emits strings that have no `(ptr, len)` header pair in `.rodata` but
are referenced by LEA instructions in `.text`. This enables extraction of:

- Hardcoded URLs passed directly to HTTP clients (Discord webhook URLs, Instagram API endpoints)
- Format strings embedded in code without a corresponding data header
- Any string literal whose address appears only in `.text`, not as a stored header in `.rodata`

### Fix 3 — First test suite

17 tests covering:

| File | Tests |
|------|-------|
| `internal/strings/extractor_test.go` | Empty input, header-pair extraction, min-length filter, non-printable filter, deduplication, out-of-range pointer |
| `internal/strings/classifier_test.go` | URL (incl. non-`https?/ftp/ws` schemes), IP, path, secret, pkgpath, plain; embedded-URL-not-URL; batch ordering; `SplitConcatenatedURLs` splits blob + passes single through |
| `internal/gopclntab/parser_test.go` | Integration: parse pclntab from test binary; assert >100 functions, non-empty GoVersion, testing.* functions present |
| `pkg/analyzer/analyzer_test.go` | Integration: full pipeline on test binary; assert TotalFunctions > 0, TotalStrings > 0 |

Run with: `go test ./...`

### Fix 4 — Type recovery returns empty slice

`Recover()` in `internal/types/recovery.go` previously returned `nil` when `.typelinks` was
absent and the fallback scan found nothing. It now returns `[]RecoveredType{}`, making callers
safe to range over the result without a nil check.

### Fix 5 — pclntab nfunc sanity check

`ScanForPclntab()` and `FindGopclntab()` now require `nfunc > 10` (lower 32 bits of the nfunc
field at header bytes [8:12]) in addition to the existing 8-byte header validation. This
filters out random data that accidentally matches the magic and pad/ptrSize constraints but
has an implausible function count.

## Known limitations (carried forward)

- **CMOVNE plain blobs** — stdlib error-message strings stored consecutively in `.rodata`
  (e.g. the P224/P256/P384/P521 nistec error series) still produce 512-byte plain blobs when
  `findLengthNearby()` cannot resolve their lengths. These do NOT appear in `--type url`
  output (classifier fix) but are visible in unfiltered output. A general-purpose split pass
  is planned for `v0.0.5-pre` (v0.0.4-pre is allocated to garble/obfuscation detection).
- **ELF not integration-tested on Linux** — the ELF loader is code-reviewed correct but the
  integration test (`os.Executable()`) runs on Windows and returns a PE binary. Run
  `go test ./...` on a Linux machine for ELF coverage.
- **x86_64 only** — ARM64 support planned for `v0.2.0`.
