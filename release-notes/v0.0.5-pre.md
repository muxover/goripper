# GoRipper v0.0.5-pre — CMOVNE Plain Blob Splitting & String Quality

**Release date:** 2026-03-09

### Install

```bash
go install github.com/muxover/goripper/cmd/goripper@v0.0.5-pre
```

Or download a pre-built binary from [Releases](https://github.com/muxover/goripper/releases/tag/v0.0.5-pre).

---

## Summary

This release eliminates the 512-byte plain blobs produced by the CMOVNE compiler pattern
for non-URL strings (stdlib error-message series, format strings). Four complementary fixes
work together: wider instruction scanning, register-aware length attribution, interval-based
blob suppression, and a 200-byte cap for stdlib-only references.

**Before:** 522 plain strings > 100 bytes in a typical nullmonitor analysis.
**After:** 40 — well below the target of < 50.

---

## What's fixed

### Fix 1 — Wider `findLengthNearby` window

**Before:** 15 instructions forward from the LEA only.

**After:** 30 instructions forward + 8 instructions backward. The backward scan
decodes forward from 64 bytes before the LEA to build instruction boundaries,
then walks the last 8 positions in reverse. This resolves length for patterns
where the MOV immediate is set in the function prologue before the LEA.

### Fix 2 — Register-aware MOV matching

**Problem:** The CMOVNE conditional-select pattern emits two MOV immediates —
one for each string's length. The first LEA's length was sometimes attributed to
the second (`MOV R8D, 34` instead of `MOV RCX, 34`).

**Fix:** `extractMovImm()` now rejects MOV instructions targeting the extended
register family (R8..R15, all widths). Only moves to the standard six registers
(AX/BX/CX/DX/SI/DI and their sized variants) are accepted. `isExtendedReg()`
is tested with the exact constant names from `golang.org/x/arch/x86/x86asm`.

### Fix 3 — Post-extraction interval blob suppression

`SuppressBlobs()` runs after `Classify()` and `SplitConcatenatedURLs()`. For
each fallback blob (string emitted by the 512-byte printable-run path), it
counts how many individually-extracted component strings start strictly inside
the blob's byte range using a binary-search over sorted VAs. If ≥ 2 component
VAs are inside the range, the blob is removed — the components are already
present in the output.

The key safety property: a blob is only suppressed when its components are
**already** in the output. Unique blobs (sole reference to that content) are
always preserved, preventing the false-negative regressions seen in v0.0.3-pre.

### Fix 4 — 200-byte stdlib-only blob cap

When every reference to a fallback blob belongs to a `runtime.*` or stdlib
function (`PackageRuntime` or `PackageStdlib`), the fallback cap is 200 bytes
instead of 512. User-code references always get the full 512-byte window.
This reduces the "radius" of stdlib noise without affecting user string extraction.

## Known limitations (carried forward)

- **ELF not integration-tested on Linux** — run `go test ./...` on Linux for ELF coverage.
- **XOR key recovery is best-effort** — only detects single-byte XOR with direct immediate operand.
- **Synthetic names from .pdata only** — ELF stripped binary fallback deferred to `v0.0.6-pre`.
- **x86_64 only** — ARM64 deferred to `v0.2.0`.
